# Knowledge Graph Visualization Tool - Implementation Plan

## Overview

This document specifies requirements and implementation details for building an interactive knowledge graph visualization tool for exploring Python codebase structure. The tool will visualize entities (classes, functions, modules) and their relationships (calls, imports, inheritance) extracted by the `kg_builder` tool.

---

## 1. Data Model Understanding

### Input Schema

The visualization tool consumes JSON files generated by `kg_builder` with the following structure:

```json
{
  "entities": {
    "<entity_id>": {
      "id": "string",              // Unique identifier (e.g., "Bio/File.py::MyClass")
      "name": "string",            // Display name
      "type": "string",            // Entity type enum
      "file_path": "string",       // Source file location
      "line_number": number,       // Definition line
      "properties": {}             // Metadata (docstring, args, decorators, etc.)
    }
  },
  "relationships": [
    {
      "source_id": "string",
      "target_id": "string",
      "type": "string",            // Relationship type enum
      "line_number": number        // Where relationship appears
    }
  ]
}
```

### Entity Types (8 types)

| Type | Description | Visual Shape Suggestion |
|------|-------------|------------------------|
| FILE | Top-level file container | Document icon / rectangle |
| MODULE | Python module | Folder icon / rounded rect |
| CLASS | Class definition | Cylinder or hexagon |
| FUNCTION | Function/method | Circle or rounded rect |
| CONSTANT | UPPERCASE assignments | Diamond with bold text |
| VARIABLE | Local/function-scoped vars | Small circle or ellipse |
| IMPORT | Import statements | External link icon |
| DECORATOR | Decorator definitions | Star or badge shape |

### Relationship Types (6 types in current output)

| Type | Description | Visual Style |
|------|-------------|--------------|
| CONTAINS | Parent contains child | Solid line, hierarchical |
| CALLS | Function invocation | Arrow with "calls" label |
| INHERITS | Class inheritance | Triangle arrow head |
| IMPORTS | Import statement | Dashed line |
| INSTANTIATES | Object creation | Dotted arrow |
| DEFINES_IN | Scope defines variable | Thin solid line |

## 2. Core Visualization Requirements

### 2.1 View Types

#### A. Force-Directed Graph View (Primary)
- **Purpose**: Explore relationships between entities across the codebase
- **Features**:
  - Nodes positioned by force simulation with entity-type-based grouping
  - Collapsible clusters for modules/files
  - Edge bundling to reduce visual clutter for dense graphs
  - Physics-based interaction (nodes repel, edges attract)

#### B. Hierarchical Tree View
- **Purpose**: Understand file/module/class containment hierarchy
- **Features**:
  - Expandable/collapsible tree structure
  - Files -> Modules -> Classes -> Functions nesting
  - Node badges showing child counts
  - Search with auto-expand to matching nodes

#### C. Module Overview View (Dashboard)
- **Purpose**: High-level codebase statistics and entry points
- **Features**:
  - Statistics panel: entity counts by type, relationship density
  - Most connected entities (hubs)
  - Top modules by complexity (lines, functions count)
  - Quick links to focus view on specific areas

#### D. Dependency Graph View
- **Purpose**: Understand import/call dependencies between modules
- **Features**:
  - Module-level aggregation (internal details collapsed)
  - Circular dependency detection and highlighting
  - Import direction arrows with external library distinction

### 2.2 Layout Algorithms

| Use Case | Recommended Algorithm | Library Support |
|----------|---------------------|-----------------|
| General exploration | Force-directed (d3-force, Cytoscape dagre) | d3.js, Cytoscape.js |
| Hierarchy visualization | Tree/indent tree | Cytoscape.js dagre extension |
| Module imports | Hierarchical edge bundling | d3.js hierarchy + bundle |
| Inheritance trees | DAG (Directed Acyclic Graph) | Cytoscape.js dagre |

## 3. User Interaction Features

### 3.1 Navigation Controls

#### Pan & Zoom
- Mouse wheel / trackpad for zooming
- Click-and-drag canvas to pan
- Double-click to center on node
- Fit-to-view button

#### Keyboard Navigation
| Shortcut | Action |
|----------|--------|
| Arrow keys | Nudge view |
| Home | Center view |
| +/- | Zoom in/out |
| Esc | Clear selection, close panels |

### 3.2 Search Functionality

#### Global Search Box
- **Live search** as user types
- Filters nodes by:
  - Entity name (fuzzy matching)
  - Entity type dropdown filter
  - File path substring match
  - Relationship type involvement
- Results highlight in graph with "Show Only" toggle
- Keyboard arrow navigation through results

#### Search Filter Options
```
[Search field] [Type dropdown] [File filter] [Show results only]
```

### 3.3 Selection & Inspection

#### Single Click on Node
- Highlights node and immediate neighbors (1-hop)
- Opens detail sidebar with entity information:
  - Name, type, file location (clickable link to line)
  - Full docstring/properties
  - Incoming/outgoing relationship counts
  - List of connected entities (expandable)

#### Double Click on Node
- "Focus" mode: centers view and collapses everything except this node + direct connections
- Useful for drilling into specific entity

#### Click and Drag Selection
- Rectangular multi-select box
- Shows aggregate info for selected group
- Bulk operations available via toolbar

### 3.4 Filter Controls

#### Entity Type Filter Panel
- Toggle checkboxes for each entity type
- Update graph in real-time
- "Show only filtered" vs "Dim unfiltered" modes

#### Relationship Type Filter Panel
- Similar toggle mechanism for edge types
- Option to show edge labels or hide for cleaner view

#### Size-based Filtering
- Collapse nodes with >N children
- Hide edges below certain importance threshold

### 3.5 Context Menu (Right-click)

| Action | Description |
|--------|-------------|
| Focus on this | Center and isolate node |
| Show path to... | Find shortest path between two entities |
| Copy entity ID | For programmatic use |
| Open in editor | Link to IDE/file viewer |
| Expand level | Expand children by N levels |
| Collapse all | Reset view to modules only |

### 3.6 Tooltip Design

- Hover over node shows: name, type, file path (truncated)
- Hover over edge shows: relationship type
- Rich tooltips for complex entities showing docstring preview
- Customizable via settings

## 4. Technical Recommendations

### 4.1 Primary Technology Stack

#### Option A: Cytoscape.js (Recommended for MVP)

**Why**: Purpose-built for graph visualization, excellent performance, rich feature set

```javascript
// Installation
npm install cytoscape cytoscape-dagre cytoscape-fcose

// Core setup
import cytoscape from 'cytoscape';
import dagre from 'cytoscape-dagre';
import fcose from 'cytoscape-fcose';

cytoscape.use(dagre);
cytoscape.use(fcose);

const elemStyle = [
  { selector: 'node[type="CLASS"]', style: { shape: 'hexagon' } },
  // ... comprehensive styling
];

const cy = cytoscape({
  container,
  elements: parseKGtoCytoscapeElements(kgData),
  style: elemStyle,
  layout: { name: 'fcose' }, // or 'dagre' for hierarchical
  zoom: 1, pan: {x:0, y:0},
});
```

**Key Extensions**:
- `cytoscape-dagre` - Hierarchical layouts
- `cytoscape-fcose` - Compound node clustering
- `cytoscape-cola` - Collision avoidance layout
- `cytoscape-expand-collapse` - Node expansion/collapse

#### Option B: D3.js + React Force Graph 2D

**Why**: Maximum customization, modern React integration

```javascript
npm install react-force-graph-2d react-simple-maps d3-hierarchy
```

**Best for**: Custom force layouts with full control over rendering

### 4.2 Frontend Framework Choice

**React (Recommended)** - Component-based architecture aligns well with:
- Reusable graph components
- State management for filters/selections
- Integration with sidebar panels

**Vanilla JS** - Simpler deployment, single HTML file option

### 4.3 Performance Optimizations

#### For Large Graphs (>1000 nodes)

| Technique | Implementation |
|-----------|---------------|
| Progressive loading | Load top-level modules first, lazy-load children on expand |
| Virtualization | Only render visible viewport nodes (d3-force has this built-in) |
| Canvas rendering | Use `canvas` renderer instead of SVG for 10k+ elements |
| Edge bundling | Hierarchical edge bundling to reduce visual complexity |
| Level of Detail | Show simplified view at low zoom, detail at high zoom |

#### Data Preprocessing
```python
# Pre-compute before visualization
- Adjacency lists for fast neighbor lookup
- Node degree statistics for importance weighting
- Connected components for modular loading
- Shortest paths cache for common query pairs
```

### 4.4 State Management

#### Reactive Approach (Vuex/Pinia or Redux)
```javascript
{
  graph: { nodes, edges, layout },
  selection: { selectedNodes: [], focusedNode: null },
  filters: { entityTypes: [], relationshipTypes: [] },
  view: { zoom, pan, currentViewMode }
}
```

## Appendix C: Known Challenges & Solutions

|-----------|----------|

| Large docstrings in sidebar | Truncate with expand preview, virtual scrolling |
| Browser memory limits | Web workers for parsing, chunked rendering |
| Finding meaningful layout | Hybrid approach: hierarchical for modules, force for internals |
